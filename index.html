Cross-Chain Journey Flow

1. Request Initiated

Smart contract on Arbitrum requests "BTC/USD" price.

2. AggregatedDataFeedStore

Request is sent to the core contract for processing.

3. Cross-Chain Messaging

Request propagated to nodes on BSC, Polygon via Hyperlane/LayerZero.

4. Oracle Nodes Fetch & ZK Prove

Each node fetches data, ZK proofs validate correctness privately.

5. zkRollup Batch & Publish

Proofs aggregated into a zkRollup block, published to Ethereum.

6. Result Returned

Verified result sent back to the requesting Arbitrum contract.

ZK Proof

A cryptographic proof that the data is valid and votes were consistent, without revealing the data or identities.

Verification

Smart contract checks the proof. Is it valid? Yes!

What this means: This proof guarantees the correctness of the data and the honesty of the voting process, while keeping the actual data and voter identities private.

Unique Offer: ZK & SchellingCoin Comparison

Feature Blocksense Chainlink Pyth
Security ZK proofs ensure data & vote correctness without revealing identities. Reputation-based system, potential for collusion. Publishers can be a central point of failure.
Decentralization Truly decentralized with ZK-protected SchellingCoin consensus. Relies on a limited set of node operators. Depends on a small set of publishers.
Cost zkRollup batching for gas-efficient publishing. Can be expensive on high gas fee networks. Generally lower cost, but model varies.
Latency Optimized for speed, with ZK for consensus. Depends on node update frequency. Very low latency, but relies on publisher speed.
Customizability Highly programmable oracles for complex needs. Limited by pre-defined feeds. Good for price data, less for custom logic.
